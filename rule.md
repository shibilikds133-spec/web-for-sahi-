# SSF Sahithyolsav 2026 Configurable Rule Source

This file defines the business rules the app must follow. It replaces hard-coded 2025 assumptions with a configurable 2026 rule model.

Agents must read this file before touching participants, registrations, settings, schedules, results, certificates, reports, transfer, storage, or database migrations.

## 1. Rule Source Policy

- 2026 is the active target year.
- 2025 rules may be used only as a reference or historical seed.
- Dates, topics, books, point values, grade thresholds, items, and restrictions must be configurable in the database.
- UI screens must load active rules/settings for the selected tenant and festival year.
- Rule functions must receive config as input and must not hard-code handbook values except for seed defaults.
- Every rule violation should return a structured error or warning with a rule id.

## 2. Festival Calendar Defaults

Default 2026 dates must be seed data, not hard-coded UI values.

Suggested configurable levels:

| Level | Default Window | Configurable |
|---|---|---|
| Family | May 01 - May 15 | Yes |
| Block / Unit | May 16 - Jun 15 | Yes |
| Sector | Jun 16 - Jun 30 | Yes |
| Campus Unit | May 15 - Jun 30 | Yes |
| Division | Jul 01 - Jul 15 | Yes |
| District | Jul 16 - Jul 31 | Yes |
| State | Aug 04 - Aug 10 | Yes |

Admin functions:

- Load default dates for festival year.
- Allow override mode.
- Validate overlapping dates.
- Store registration open and close windows separately where needed.
- Audit date changes.

## 3. Categories

Categories must be database/config driven.

Default categories:

- LP: Lower Primary.
- UP: Upper Primary.
- HS: High School.
- HSS: Higher Secondary.
- JR: Junior.
- SR: Senior.
- GN: General.
- CA: Campus.
- CG: Campus Girls / Parallel where applicable.

Rules:

- Category detection should be based on DOB/class/config.
- Post-HS religious students must compete in JR/SR according to age/config.
- HSS restriction for post-HS religious students must be enforced when active.
- Campus Girls Parallel must require recognized board/university proof when active.

## 4. Registration Rules

Default registration rules:

| Rule ID | Rule | Behavior |
|---|---|---|
| REG-01 | Max item count | Max 4 non-General items by default. Configurable. |
| REG-02 | Category match | Participant age/class/category must match item category. |
| REG-03 | Item level availability | Item must be enabled for current festival level. |
| REG-04 | Same unit/team | Participant must compete from allowed unit/team only. |
| REG-05 | Dual team block | Same participant cannot compete from multiple teams in the same festival. |
| REG-06 | Dual category block | Dual category participation blocked except configured exceptions such as General. |
| REG-07 | Group member count | Min/max members must match item config. |
| REG-08 | Group member uniqueness | Same participant cannot appear twice in the same group. |
| REG-09 | Plagiarism ban | Active ban blocks registration until ban end date. |
| REG-10 | Registration window | Registration allowed only inside open/close dates. |
| REG-11 | Duplicate registration | Same participant and same item cannot be registered twice. |
| REG-12 | Daf/Arabana restriction | Daf/Arabana item restrictions must be enforced from item config. |
| REG-13 | Regional language restriction | Restricted language/dialect warnings must show for configured items. |

Rule output shape:

```ts
type RuleResult = {
  ruleId: string;
  severity: 'error' | 'warning' | 'info';
  message: string;
  metadata?: Record<string, unknown>;
};
```

## 5. Group Item Rules

- Group items start level must be configurable.
- Sector friendly match behavior must be configurable.
- Division result publishing can lock group members.
- Member changes after lock must be blocked unless a super-admin override exists.
- Group member list must transfer upward with the team.
- Daf, Arabana, Sanghaganam, Khavali, Maulid, Mappilappattu, and similar items must get their member counts and special restrictions from item config.

## 6. Chest Number Rules

- One participant gets one chest number for the festival/level.
- Same participant keeps the same chest number across items.
- Category-wise series is the default: `LP-001`, `UP-001`, `HS-001`.
- Series prefix, start number, padding, and manual override must be configurable.
- Manual override must check duplicates.
- Chest number changes must be audited.

## 7. Schedule Rules

- Venue conflicts must be blocked.
- Configurable buffer time, default 15 minutes.
- Sector and Division festival duration default maximum: 2 days.
- Duration limit must be configurable by festival level.
- Code letters must be generated by random draw and stored.
- Check-in must support code-letter order and QR/manual flow.

## 8. Judge And Mark Rules

- Minimum judges default: 3, configurable.
- Criteria total default: 100, configurable.
- Marks can be saved as draft.
- Final submit must lock marks unless authorized reopen is implemented.
- Results should be ready within configured time, default 30 minutes after competition.

## 9. Result Rules

Default grade thresholds:

| Grade | Threshold |
|---|---|
| A+ | 90% |
| A | 75% |
| B | 60% |
| C | 50% |

Default points:

| Result | Points |
|---|---|
| 1st Place | 10 |
| 2nd Place | 7 |
| 3rd Place | 5 |
| A+ Grade | 4 |
| A Grade | 3 |
| B Grade | 2 |
| C Grade | 1 |
| Participation | 0 |

Rules:

- Judge average decides score.
- Tie break uses highest main criteria when configured.
- Exact tie can produce shared rank when configured.
- Less than 3 teams: grade only by default; no rank points. Configurable.
- Result publish/unpublish must write audit logs.
- Point table should be recalculated from a ledger, not hand-edited totals.

## 10. Certificate Rules

- All grade holders are eligible for certificates by default.
- Certificate template must be configurable per year/level.
- Certificate must include year, level, category, item, participant, chest number, grade/rank when applicable, and verification id.
- Bulk generation should run through a job/queue.
- Generated certificate files should be stored through storage service.
- Public verification should not expose private participant data beyond certificate validity and published details.

## 11. Transfer Rules

- Transfer direction is upward only:
  - Unit -> Sector.
  - Sector -> Division.
  - Division -> District.
  - District -> State.
- Qualified participants/groups must be selected by result/rank rules.
- Duplicate detection must use at least name + DOB + category + tenant/festival context.
- Transfer package can support QR, JSON, Excel, or sync code.
- QR/sync code expiry default: 10 minutes, configurable.
- Festival year validation is mandatory.
- Group member lock after Division result publish must be enforced when active.

## 12. Storage Rules

- Participant photos, import files, reports, and certificates must have file metadata in the database.
- Storage provider must be configurable: Supabase Storage now, R2 later.
- UI must call storage service only.
- Frontend must never include service keys or R2 secret keys.
- Bucket changes are rule-bound changes. Agents must read `plan.md` bucket migration section before touching them.
- Logical bucket ids must be stable even when provider bucket names change.
- Private files must use signed URLs or controlled API routes.
- Public buckets are allowed only for non-sensitive assets.
- Certificate files should be private by default; public verification must expose controlled certificate data only.
- Import files and transfer packages must never be public.
- Bucket policies, allowed MIME types, max file size, and object key format must be configurable per logical bucket.

Default logical buckets:

| Logical Bucket | Sensitive | Public Allowed |
|---|---|---|
| `participant-photos` | Yes | No |
| `import-files` | Yes | No |
| `reports` | Yes | No |
| `certificates` | Yes | Controlled verification only |
| `public-assets` | No | Yes |
| `transfer-packages` | Yes | No |

## 13. Appeals And Complaints

- Published results may open an appeal window.
- Appeal window duration must be configurable.
- Complaint status should be tracked.
- Resolution should be auditable.

## 16. Agent Rule

When implementing any feature:

1. Read `plan.md`.
2. Read `project.md`.
3. Read this `rule.md`.
4. State which rule ids apply.
5. Implement only according to the relevant rules.
6. If a needed rule is missing or unclear, add it here as configurable before coding behavior.

## 17. NestJS Rule

NestJS may be used as the future backend API layer, but it must not change the rule source.

- Rules remain configurable by festival year, tenant, level, and active rule set.
- NestJS must call shared backend rule services, not duplicate ad hoc rule logic inside route handlers.
- Frontend services may call NestJS through an API provider, but UI screens must not call NestJS directly.
- Supabase/Postgres rules and NestJS rules must produce the same validation result during migration.
- Sensitive work such as storage signed URLs, sync validation, report generation, and certificate generation should move to NestJS before public exposure.
